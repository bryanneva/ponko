# OpenTelemetry Naming Conventions

This document defines naming conventions for all OTel instrumentation in ponko. Consistency here ensures spans, metrics, and logs are queryable and meaningful in Grafana Cloud.

## Span Naming

Spans follow the pattern `{category}.{operation}`:

| Category | Pattern | Examples |
|----------|---------|----------|
| HTTP inbound | `{METHOD} {route}` | `GET /api/workflows/recent`, `POST /api/workflows/start` |
| Job execution | `job.{worker}` | `job.plan`, `job.execute`, `job.synthesize`, `job.slack_reply` |
| LLM calls | `llm.{provider}.{op}` | `llm.anthropic.messages`, `llm.anthropic.messages_stream` |
| MCP tool calls | `mcp.{server}.{tool}` | `mcp.knowledge.search`, `mcp.slack.send_message` |

HTTP spans are auto-generated by `otelhttp` middleware using the standard `{METHOD} {route}` format.

## Resource Attributes

Set on the TracerProvider at init — attached to every span and metric:

| Attribute | Source | Example |
|-----------|--------|---------|
| `service.name` | `OTEL_SERVICE_NAME` env var | `ponko` |
| `deployment.environment` | `FLY_APP_NAME` presence or explicit env | `production`, `development` |
| `service.version` | Build-time or `FLY_IMAGE_REF` | `v1.2.3` |

## Span Attributes

Per-span attributes for queryability:

| Attribute | Type | Used On | Description |
|-----------|------|---------|-------------|
| `workflow.id` | string | Job spans | UUID of the originating workflow — links all jobs in a chain |
| `workflow.type` | string | Job spans | e.g. `slack_mention`, `api_trigger` |
| `job.kind` | string | Job spans | River worker kind, e.g. `plan`, `execute` |
| `llm.model` | string | LLM spans | Model identifier, e.g. `claude-sonnet-4-20250514` |
| `llm.token.input` | int | LLM spans | Input token count |
| `llm.token.output` | int | LLM spans | Output token count |
| `mcp.server` | string | MCP spans | MCP server name |
| `mcp.tool` | string | MCP spans | MCP tool name |
| `error` | bool | Any span | Set true on error via `span.SetStatus(codes.Error)` |

## Go Attribute Patterns

Use `attribute.String()` / `attribute.Int64()` for typed attributes:

```go
span.SetAttributes(
    attribute.String("workflow.id", workflowID),
    attribute.String("job.kind", "plan"),
    attribute.Int64("llm.token.input", int64(inputTokens)),
)
```

For error recording:

```go
if err != nil {
    span.RecordError(err)
    span.SetStatus(codes.Error, err.Error())
}
```

## Metrics

Custom metrics use the `ponko.` prefix:

| Metric | Type | Description |
|--------|------|-------------|
| `ponko.workflow.completed` | Counter | Workflows that completed successfully |
| `ponko.workflow.failed` | Counter | Workflows that failed |

Runtime metrics (Go memory, GC, goroutines) are collected automatically via `runtime.Start()`.

## Workflow Trace Context Propagation

Trace context flows through the River job chain via a `TraceContext` string field on job args structs:

1. **HTTP request arrives** → otelhttp middleware starts a root span
2. **Job enqueued** → `otel.InjectTraceContext(ctx)` serializes the W3C `traceparent` into the args
3. **Worker picks up job** → `otel.ExtractTraceContext(args.TraceContext)` restores the parent context
4. **Worker starts child span** → linked to the original HTTP request trace
5. **Worker enqueues next job** → injects current span context into downstream args

This gives a single trace spanning: HTTP request → receive → plan → execute → process → synthesize → respond/slack_reply.

Jobs not part of workflow chains (`SchedulerTick`, `ProactiveMessage`) do not carry trace context.
